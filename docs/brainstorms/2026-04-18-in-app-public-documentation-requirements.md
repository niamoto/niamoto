# In-App Public Documentation

## Problem Frame

Niamoto now has a clearer public documentation tree under `docs/01-*` through
`docs/09-*`, but the app shell still treats documentation as either a dead-end
Help menu or an API-only surface. That leaves a mismatch between the public
docs that describe the product and the desktop/web app that should help users
discover them in context.

The requirement is broader than the earlier desktop user-guide integration
idea. The app should expose the public numbered documentation corpus through a
single in-app Documentation entry point, while keeping API docs separate. The
experience should remain simple in the shell, searchable in the command palette,
and automatic enough that adding a future section such as `docs/10-foo/`
appears without hand-maintained routing work after the next build.

## Visual Map

| Layer | Audience | In-app behavior | Source of truth |
| --- | --- | --- | --- |
| Documentation home | End user, evaluator, contributor | One `Documentation` entry in the app opens an editorial landing page | Generated from public numbered docs |
| Section entry | Reader entering a topic area | Each top-level numbered section uses its `README.md` as the editorial landing page | `docs/01-*` to `docs/09-*`, and future numbered sections |
| Page discovery | Reader looking for a specific topic | Search results appear in the command palette under a grouped `Documentation` section | Page titles and intertitles extracted from public docs |
| API reference | Developer | Separate surface, not merged into the main Documentation help center | Existing API docs implementation |

## Requirements

**App Entry And Shell**
- R1. The app must expose a single top-level `Documentation` entry point in the
  shell instead of multiple top-level documentation destinations.
- R2. Opening `Documentation` must land on an editorial home page inside the
  app, not directly on the first article.
- R3. The command palette must expose an `Open Documentation` action.
- R4. Documentation pages and sections must also be discoverable inside the
  command palette results, grouped under a dedicated `Documentation` result
  group.

**Corpus Scope**
- R5. The in-app documentation corpus must be built from the public numbered
  documentation sections, not only the desktop user guide.
- R6. The source corpus must include public `.md` and `.rst` pages found under
  top-level numbered docs directories such as `docs/01-*` through `docs/09-*`,
  and future matching directories such as `docs/10-*`.
- R7. The first pass must explicitly exclude autogenerated API reference content
  under `docs/06-reference/api/` from the embedded Documentation experience.
- R8. Internal material such as `docs/plans/`, `docs/brainstorms/`,
  `docs/ideation/`, and `docs/_archive/` must not be exposed in the in-app
  Documentation corpus.

**Automatic Discovery**
- R9. Adding a new top-level public docs section that matches the numbered
  convention and includes a section `README.md` must make that section appear
  automatically in the in-app Documentation after the next build.
- R10. The system must support a simple per-file or per-section opt-out marker
  so a public numbered page can be excluded from the in-app corpus without
  custom code changes.
- R11. The section structure shown in the app must treat each numbered
  section's `README.md` as the editorial entry point, then complete that section
  with the other pages in the same docs subtree.

**Search And Retrieval**
- R12. Documentation search in the app must index section titles, page titles,
  and intertitles from the source docs.
- R13. The first pass must not require full-text indexing of all body
  paragraphs.
- R14. Search results must open the matching section or article directly inside
  the in-app Documentation surface.

**Separation From API Docs**
- R15. User-facing Documentation and API documentation must remain distinct in
  naming, routing, and search behavior.
- R16. The existing API docs surface may remain available in parallel, but it
  must not be the default destination behind generic `Documentation` affordances.

**Content And Assets**
- R17. The public docs remain the editorial source of truth for the in-app
  content. The app must consume generated artifacts from those docs rather than
  maintain a second handwritten copy.
- R18. Public screenshots or other media used by pages that appear in the
  in-app Documentation must live in a stable public docs asset location, not in
  planning-only directories.

## Success Criteria

- A user can open `Documentation` inside the app and browse the public numbered
  documentation corpus from a clear home page.
- The shell stays simple: one Documentation entry in the UI, not a proliferation
  of top-level docs links.
- The command palette can find docs entries by section title, page title, and
  intertitles, and those results feel integrated rather than bolted on.
- Adding a future section such as `docs/10-foo/README.md` requires no new
  manual route registration for it to appear after rebuild.
- API docs remain accessible but are clearly separated from the main
  Documentation experience.
- Internal planning and archive material do not leak into the in-app docs.

## Scope Boundaries

- No redesign of the generated API reference in this pass.
- No requirement that the app shell expose every docs section as a dedicated
  navigation item.
- No full-text search across every paragraph in the first pass.
- No exposure of internal planning, brainstorming, ideation, or archive trees.
- No requirement yet for contextual help buttons inside every module screen.

## Key Decisions

- One Documentation entry in the shell is better than a large docs menu.
- The command palette should act as the secondary discovery surface for docs.
- The embedded corpus should follow the public numbered docs convention rather
  than hand-curated article lists.
- The structure should be rebuilt from section `README.md` pages plus the rest
  of each section subtree, not from a Sphinx-only navigation parser.
- API docs should stay separate for now, especially the autogenerated
  `docs/06-reference/api/` subtree.
- Automatic inclusion is the default, but the system needs a simple opt-out so
  one accidental page does not force code changes.
- Search should stop at titles and intertitles in v1; that is enough to make
  the corpus discoverable without immediately carrying full-text complexity.

## Dependencies / Assumptions

- Public numbered docs continue to follow a stable top-level naming convention
  such as `01-*`, `02-*`, and future `10-*`.
- Each public section continues to provide a meaningful `README.md`.
- The build pipeline can generate a manifest or equivalent artifact from the
  docs tree before packaging.
- Public screenshots currently sourced from `docs/plans/caps/` can be moved to
  a stable public asset directory before the in-app corpus depends on them.
